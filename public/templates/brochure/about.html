Design Patterns

- Design patterns employed -


Front Controller


The front controller pattern is where you have a single entrance point for your web application that handles all of the requests. This code is responsible for loading all of the dependencies, processing the request and sending the response to the browser. The front controller pattern can be beneficial because it encourages modular code and gives you a central place to hook in code that should be run for every request such as input sanitization.

In this project the entire web api interface is handled by a single controller: \Core\Controller\ApiController


Registry


To implement a central storage for objects often used throughout the application, this is typically implemented using an abstract class with only static methods (or using the Singleton pattern).

For this project the registry (\Core\Registry) provides global access to the Doctrine Entity Manager and all Service classes.


Dependency Injection


To implement a loosely coupled architecture in order to get better testable, maintainable and extendable code.

For this project the registry (\Core\Registry) provides dependency injection of the Doctrine Entity Manager into the service classes to ensure an architecture where test code can provide either a mock entity manager or one pointed to a test database.




Best Practice Implementation

- Implementation of selected best practices -


Black-box / De-coupling


Services black-boxed and de-coupled to allow for easy replacement of services without affecting the rest of the codebase.


Separation of Concerns


For this project separation was implemented by providing isolation of Services, API and Input Validation.


Thin Controller


By keeping the controller thin (in this case only handling validation and service routing) and keeping the business logic in the service layer provides for a central location of business logic without the API endpoint being affected by service changes and business logic changes.




API Design

- Implementation of web service API -


Front Controller


The Front Controller patterns allow for a simplified and consistent api that minimizes maintenance and debugging on the API layer. This also assists in encouraging a standardized service interface for all service classes.


API Versioning


By versioning the API and having the version of the API specified in the URL assists in provided dedicated space for different version API that could have breaking changes without affecting existing systems that will need time to migrate to a newer version.


JSON POST


Although many suggest a rest interface using the various http methods such as GET and PUT I have chosen to implement a pure POST architecture using JSON as the data format. The advantage of this is once again a simpler API that is easier to maintain and also provides the possibility of easier access from a client as there needs to be no switching between methods. It also assists in ease of debugging as there is only one end-point and one method of data transfer.
